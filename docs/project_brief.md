# プロジェクト理解メモ

## 位置づけ

このプロジェクトは、暗号資産デリバティブ商品を「売れる商品」としてではなく、「顧客に説明できるリスク構造」として理解するためのコードベースです。

想定する実務文脈は、富裕層や機関投資家に対して、BTC/ETH/USDT/USDCを参照するDual Currency Investment、Stable-Return型商品、レンジ型、バリア型、オートコーラブル型の商品を提案する場面です。

営業・RMの観点では、重要なのは利回りの高さではありません。重要なのは、顧客がどのリスクを引き受け、その対価としてどの程度の期待収益を得るのかを、定量的に説明できることです。

## 最初に実装する商品

### Dual Currency Investment

最初のMVPではDCIを中心にします。理由は、暗号資産の仕組商品の中で構造が比較的単純でありながら、顧客説明で重要な論点がほぼ全部入っているためです。

USDT/USDC建てDCI:

- 顧客はstablecoinを預ける
- BTCがストライク以上ならstablecoinで元本と利回りを受け取る
- BTCがストライク未満ならBTCに転換される
- 実質的にはcash-secured put sellingに近い

BTC建てDCI:

- 顧客はBTCを預ける
- BTCがストライク以下ならBTCで元本と利回りを受け取る
- BTCがストライク超ならstablecoinに転換される
- 実質的にはcovered call sellingに近い

## 数学的に見るべきもの

商品の説明で最低限必要な統計量は次の通りです。

- 期待値: 平均的に顧客に残る価値
- 期待リターン: 元本に対する平均損益率
- 損失確率: 満期時価値が初期元本を下回る確率
- 転換確率: 不利な受渡条件に入る確率
- VaR: 指定信頼水準での損失
- CVaR: 悪いケースだけを見た平均損失
- benchmark gap: stablecoin保有やBTC単純保有との差

ここでのポイントは、表面利回りAPRと期待リターンは同じではないということです。APRは顧客に見えるクーポンですが、期待リターンは不利な転換や価格変動を含めた後の平均値です。

## Binance APIの使い方

この段階ではAPIキー不要の公開データだけを使います。

- `GET /fapi/v1/klines`: BTCUSDTなどの価格系列
- `GET /fapi/v1/fundingRate`: funding rate history
- `GET /fapi/v1/exchangeInfo`: 取引可能symbolや取引ルール

価格系列からヒストリカル・ボラティリティを推定し、Monte Carloで満期価格分布を作ります。funding rateは、次の段階でStable-Return型やbasis/funding型商品の収益源を分析するために使います。

## 今後追加するモジュール

1. Range Accrual

BTCが一定レンジ内にいる日数に応じてクーポンが発生する商品です。顧客は実質的にボラティリティを売っています。

2. Barrier Reverse Convertible

BTCが一定バリアを割らなければクーポンを得ます。バリアを割ると元本毀損やBTC受渡が発生します。

3. Basis/Funding Stable-Return

BTC現物ロングとperpetual shortなどを組み合わせ、fundingやbasisを収益源にする商品です。方向性リスクを抑えても、funding反転、清算、流動性、取引所リスクが残ります。

4. Suitability Dashboard

顧客向け説明に必要な、最悪ケース、ストレスシナリオ、商品別比較、リスク説明文を自動生成するCLIまたは小さなWeb UIを作ります。

## 販売時の期待値管理

顧客に説明すべき表現は、次のようなものです。

「この商品は年率12%の利回りを表示していますが、これは元本保証の利回りではありません。BTCがストライクを下回る場合、満期時にBTCで受け取るため、時価評価では損失が出る可能性があります。シミュレーション上の損失確率、転換確率、VaRは以下の通りです。」

この一文をコードで裏付けるのが、このプロジェクトの目的です。
